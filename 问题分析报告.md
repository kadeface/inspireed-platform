# 教案内容显示问题分析报告

## 问题描述
本地 dev 分支教案有具体内容，而 `remotes/origin/production-deploy` 分支无法显示具体内容。

## 根本原因

### 核心问题
`frontend/src/services/api.ts` 中 API 地址检测逻辑的**优先级差异**导致 production-deploy 分支在生产环境中可能使用了错误的 API 地址。

### 代码差异对比

#### dev 分支（正常工作）
```typescript
function getApiBaseUrl(): string {
  // 1. 先动态检测环境（CloudStudio、生产环境等）
  const hostname = window.location.hostname
  // ... 动态检测逻辑 ...
  
  // 2. 然后才检查环境变量（作为备选）
  if (import.meta.env.VITE_API_BASE_URL) {
    // ... 处理环境变量 ...
  }
}
```

#### production-deploy 分支（有问题）
```typescript
function getApiBaseUrl(): string {
  // 1. 优先使用环境变量（如果已配置）
  if (import.meta.env.VITE_API_BASE_URL) {
    return envApiUrl  // 直接返回，不再进行动态检测
  }
  
  // 2. 如果没有环境变量，才使用动态检测
  // ... 动态检测逻辑 ...
}
```

### 问题场景

1. **Docker 构建配置**：
   ```yaml
   # docker-compose.prod.yml
   frontend:
     build:
       args:
         VITE_API_BASE_URL: ${VITE_API_BASE_URL:-}  # 如果未设置，传入空字符串
   ```

2. **Dockerfile 默认值**：
   ```dockerfile
   ARG VITE_API_BASE_URL=http://localhost:8000/api/v1  # 默认值
   ENV VITE_API_BASE_URL=$VITE_API_BASE_URL
   ```

3. **问题发生流程**：
   - 生产环境构建时，`VITE_API_BASE_URL` 未设置或为空
   - Dockerfile 使用默认值 `http://localhost:8000/api/v1`
   - production-deploy 分支优先使用这个环境变量
   - 前端尝试连接到 `localhost:8000`，而不是实际的后端地址
   - **结果**：无法获取教案内容或获取到错误的数据

### 为什么 dev 分支正常？

- dev 分支优先动态检测，即使环境变量配置错误，也能通过动态检测找到正确的后端地址
- 更适合开发和测试环境，具有更好的容错性

## 解决方案

### 方案一：修改 production-deploy 分支的 API 检测逻辑（推荐）

将 production-deploy 分支的 API 检测逻辑改为与 dev 分支一致，优先动态检测，环境变量作为备选。

**优点**：
- 提高容错性，即使环境变量配置错误也能正常工作
- 与 dev 分支保持一致，减少维护成本
- 适合各种部署环境（本地、CloudStudio、生产环境）

**实施步骤**：
1. 将 dev 分支的 `frontend/src/services/api.ts` 中的 `getApiBaseUrl()` 函数逻辑应用到 production-deploy 分支
2. 确保环境变量检查在动态检测之后

### 方案二：确保生产环境正确配置环境变量

在构建脚本或 CI/CD 中正确设置 `VITE_API_BASE_URL`。

**优点**：
- 保持 production-deploy 分支的当前逻辑
- 通过配置管理确保正确性

**缺点**：
- 需要确保所有部署环境都正确配置
- 配置错误时无法自动恢复

**实施步骤**：
1. 在构建脚本中设置正确的 `VITE_API_BASE_URL`
2. 确保 Docker Compose 或 CI/CD 配置中正确传递环境变量
3. 验证构建后的前端应用能正确连接到后端

### 方案三：改进 Dockerfile 默认值处理

如果环境变量为空，不设置默认值，让前端代码使用动态检测。

**优点**：
- 保持 production-deploy 分支的当前逻辑
- 通过改进默认值处理提高容错性

**实施步骤**：
1. 修改 `frontend/Dockerfile`，如果 `VITE_API_BASE_URL` 为空，不设置环境变量
2. 让前端代码的 `getApiBaseUrl()` 函数使用动态检测

## 验证方法

1. **检查 API 地址**：
   - 在浏览器控制台查看 `🚀 [API] 最终使用的 API 基础地址`
   - 确认地址是否正确指向后端

2. **检查网络请求**：
   - 打开浏览器开发者工具的 Network 标签
   - 查看教案相关的 API 请求
   - 确认请求 URL 是否正确
   - 检查响应数据是否包含 content

3. **检查后端日志**：
   - 查看后端日志，确认是否有教案请求
   - 检查返回的数据格式

4. **对比测试**：
   - 在 dev 分支和 production-deploy 分支分别测试
   - 对比 API 地址和响应数据

## 相关文件

- `frontend/src/store/lesson.ts` - 教案 Store（有小的差异，但不影响读取）
- `docker/docker-compose.prod.yml` - Docker 生产配置
- `frontend/Dockerfile` - 前端 Dockerfile

## 总结

问题的根本原因是 **API 地址检测逻辑的优先级差异**。production-deploy 分支优先使用环境变量，如果环境变量配置错误，会导致无法正确连接到后端，从而无法获取教案内容。

**推荐使用方案一**，将 production-deploy 分支的 API 检测逻辑改为与 dev 分支一致，提高容错性和一致性。
