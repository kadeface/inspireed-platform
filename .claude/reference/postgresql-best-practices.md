## 1. When to Use PostgreSQL

### Ideal Use Cases

- **Enterprise applications**: Complex business logic, reliable data integrity
- **High concurrent writes**: Multiple writers simultaneously
- **Complex data relationships**: Advanced JOIN capabilities, CTEs, window functions
- **Geospatial data**: PostGIS extension for GIS applications
- **Full-text search**: Built-in full-text search capabilities
- **JSON/JSONB data**: Native JSON support with indexing
- **Time-series data**: Specialized functions and data types
- **Multi-tenant applications**: Schema-per-tenant or row-level security

### When NOT to Use PostgreSQL

- **Embedded applications**: Need zero-configuration file database (use SQLite)
- **Simple CRUD apps**: Overkill for small, simple applications
- **Resource-constrained environments**: PostgreSQL has higher memory/CPU requirements

### Key Characteristics

| Feature | Value |
|---------|-------|
| Concurrency | MVCC (Multi-Version Concurrency Control) |
| Max database size | Unlimited |
| Max row size | 1.6 TB |
| Concurrent writers | Unlimited |
| ACID compliant | Yes |
| Extensions | JSON, UUID, PostGIS, pgcrypto, etc. |

## 2. Schema Design

### Primary Keys

```sql
-- Recommended: SERIAL or BIGSERIAL (auto-increment)
CREATE TABLE habits (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL
);

-- For distributed systems: UUID
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

CREATE TABLE habits (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name TEXT NOT NULL
);

-- IDENTITY column (PostgreSQL 10+, recommended)
CREATE TABLE habits (
    id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL
);

-- Composite primary key
CREATE TABLE completions (
    habit_id INT NOT NULL,
    completed_date DATE NOT NULL,
    status TEXT NOT NULL DEFAULT 'completed',
    PRIMARY KEY (habit_id, completed_date)
);
```

### Foreign Keys

```sql
-- Foreign keys are ENABLED by default
CREATE TABLE completions (
    id SERIAL PRIMARY KEY,
    habit_id INT NOT NULL,
    completed_date DATE NOT NULL,
    FOREIGN KEY (habit_id) REFERENCES habits(id) ON DELETE CASCADE
);
```

**Cascading actions**:

| Action | Behavior |
|--------|----------|
| `NO ACTION` | Reject if child rows exist (default) |
| `CASCADE` | Delete/update child rows |
| `SET NULL` | Set foreign key to NULL |
| `SET DEFAULT` | Set foreign key to default value |
| `RESTRICT` | Like NO ACTION but immediate |

### Table Constraints

```sql
CREATE TABLE habits (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    color TEXT DEFAULT '#10B981',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    archived_at TIMESTAMPTZ,

    -- Check constraint
    CHECK (length(name) > 0),
    CHECK (color ~ '^#[0-9A-Fa-f]{6}$'),

    -- Unique constraint
    CONSTRAINT unique_name UNIQUE (name)
);

CREATE TABLE completions (
    id SERIAL PRIMARY KEY,
    habit_id INT NOT NULL,
    completed_date DATE NOT NULL,
    status TEXT NOT NULL DEFAULT 'completed',

    FOREIGN KEY (habit_id) REFERENCES habits(id) ON DELETE CASCADE,
    CONSTRAINT unique_habit_date UNIQUE (habit_id, completed_date),
    CONSTRAINT valid_status CHECK (status IN ('completed', 'skipped'))
);
```

### Unlogged Tables

```sql
-- Use for temporary data, faster but not crash-safe
CREATE TABLE temp_events (
    id SERIAL PRIMARY KEY,
    event_data JSONB
) UNLOGGED;
```

**When to use**:
- Temporary/cache data
- Data that can be recreated
- Write-heavy, read-seldom tables

**When to avoid**:
- Critical business data
- Data requiring crash recovery

## 3. Data Types

### Numeric Types

```sql
-- Integer types
CREATE TABLE example (
    small_int SMALLINT,      -- 2 bytes: -32,768 to 32,767
    integer INT,             -- 4 bytes: -2.1B to 2.1B
    big_int BIGINT,          -- 8 bytes: huge range
    serial SERIAL,           -- Auto-increment INT
    bigserial BIGSERIAL      -- Auto-increment BIGINT
);

-- Floating point
CREATE TABLE example (
    real_num REAL,           -- 4 bytes, single precision
    double_num DOUBLE PRECISION,  -- 8 bytes, double precision
    decimal_num DECIMAL(10,2),  -- Exact precision: 10 digits, 2 after decimal
    numeric_num NUMERIC(20,5)    -- Arbitrary precision
);

-- Use DECIMAL/NUMERIC for money!
CREATE TABLE products (
    price NUMERIC(10,2) NOT NULL  -- Never use FLOAT for money!
);
```

### Date/Time Types

```sql
CREATE TABLE events (
    id SERIAL PRIMARY KEY,

    -- Date only (no time)
    event_date DATE NOT NULL,      -- 2025-01-15

    -- Time only (no date)
    event_time TIME NOT NULL,      -- 14:30:00

    -- Timestamp with time zone (RECOMMENDED)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Timestamp without time zone
    updated_at TIMESTAMP
);

-- Query examples
SELECT * FROM events WHERE event_date = '2025-01-15';
SELECT * FROM events WHERE created_at >= '2025-01-01' AND created_at < '2025-02-01';

-- Date functions
SELECT NOW();                              -- Current timestamp
SELECT CURRENT_DATE;                       -- Current date
SELECT event_date + INTERVAL '1 week';     -- Add 1 week
SELECT event_date - INTERVAL '7 days';     -- Subtract 7 days
```

**Benefits**: Timezone-aware, extensive date functions, efficient storage.

### Boolean Handling

```sql
-- PostgreSQL has native BOOLEAN type
CREATE TABLE habits (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE
);

-- TRUE, FALSE, or NULL
INSERT INTO habits (name, is_active) VALUES ('Exercise', TRUE);
SELECT * FROM habits WHERE is_active = TRUE;
```

### JSON/JSONB Storage

```sql
-- JSON: stores as text, preserves whitespace and order
-- JSONB: stored in decomposed binary format, indexed, faster (RECOMMENDED)
CREATE TABLE habits (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    settings JSONB  -- Binary JSON
);

INSERT INTO habits (name, settings)
VALUES ('Exercise', '{"reminder_time": "09:00", "notifications": true}'::jsonb);

-- Query JSONB
SELECT
    name,
    settings->>'reminder_time' as reminder,
    settings->'notifications' as notifications_enabled
FROM habits
WHERE settings->>'notifications' = 'true';

-- JSONB operators
SELECT * FROM habits WHERE settings @> '{"notifications": true}';   -- Contains
SELECT * FROM habits WHERE settings ? 'reminder_time';             -- Has key
SELECT * FROM habits WHERE settings->>'reminder_time' = '09:00';   -- Value match

-- Create GIN index for JSONB queries
CREATE INDEX idx_habits_settings ON habits USING GIN (settings);
```

### Arrays

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    tags TEXT[]              -- Array of text
);

INSERT INTO users (name, tags)
VALUES ('Alice', ARRAY['developer', 'postgres', 'python']);

SELECT * FROM users WHERE 'postgres' = ANY(tags);
SELECT * FROM users WHERE tags @> ARRAY['postgres'];  -- Contains all
```

### Custom Enum Types

```sql
-- Create enum type
CREATE TYPE habit_status AS ENUM ('active', 'archived', 'deleted');

CREATE TABLE habits (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    status habit_status NOT NULL DEFAULT 'active'
);

-- Note: Enum values are case-sensitive
-- Always use lowercase for consistency with APIs
```

### Enum Type Management

```sql
-- Add new value to enum (can only add at end)
ALTER TYPE habit_status ADD VALUE 'suspended';

-- Rename enum type
ALTER TYPE habit_status RENAME TO habit_state;

-- Drop enum type (only if not used by any table)
DROP TYPE IF EXISTS habit_status;

-- List all enum values
SELECT enumlabel FROM pg_enum WHERE enumtypid = 'habit_status'::regtype;
```

**Important:**
- PostgreSQL ENUM values are case-sensitive
- You cannot remove enum values (only add new ones)
- Changing existing enum values requires creating new type and migrating data
- Always use lowercase values for JSON API compatibility

## 4. Indexing

### When to Index

- Columns in `WHERE` clauses
- Columns in `JOIN` conditions
- Columns in `ORDER BY` clauses
- Foreign key columns
- Frequently filtered columns

### Index Types

```sql
-- B-tree index (default, good for equality and range queries)
CREATE INDEX idx_habits_name ON habits(name);

-- Composite index (column order matters!)
CREATE INDEX idx_completions_habit_date ON completions(habit_id, completed_date);

-- Unique index
CREATE UNIQUE INDEX idx_habits_name_unique ON habits(name);

-- Partial index (indexes subset of rows)
CREATE INDEX idx_active_habits ON habits(name) WHERE archived_at IS NULL;

-- Expression index
CREATE INDEX idx_habits_lower_name ON habits(lower(name));

-- GIN index (for arrays, JSONB, full-text search)
CREATE INDEX idx_habits_settings ON habits USING GIN (settings);

-- GiST index (for geometric data, range types)
CREATE INDEX idx_events_date_range ON events USING GIST (event_date_range);
```

### Composite Index Column Order

The order of columns in a composite index matters:

```sql
CREATE INDEX idx_completions ON completions(habit_id, completed_date);

-- Uses index (habit_id is leftmost)
SELECT * FROM completions WHERE habit_id = 1;

-- Uses index (both columns, in order)
SELECT * FROM completions WHERE habit_id = 1 AND completed_date = '2025-01-15';

-- Uses index partially (only habit_id)
SELECT * FROM completions WHERE habit_id = 1 ORDER BY completed_date;

-- Does NOT use index (completed_date is not leftmost)
SELECT * FROM completions WHERE completed_date = '2025-01-15';
```

### Covering Indexes (INCLUDE)

```sql
-- Include non-key columns to avoid table lookup
CREATE INDEX idx_completions_covering ON completions(habit_id, completed_date)
INCLUDE (status);

-- This query is fully satisfied by the index
SELECT completed_date, status FROM completions WHERE habit_id = 1;
```

### Concurrent Index Creation

```sql
-- Create index without blocking writes (takes longer)
CREATE INDEX CONCURRENTLY idx_habits_name ON habits(name);
```

### Index Trade-offs

| Benefit | Cost |
|---------|------|
| Faster reads | Slower writes |
| Faster ORDER BY | More disk space |
| Faster JOINs | Memory overhead |
| Unique enforcement | Update overhead |

**Rule of thumb**: Index only columns used in WHERE, JOIN, ORDER BY clauses.

## 5. Query Optimization

### EXPLAIN ANALYZE

```sql
-- Analyze query execution plan
EXPLAIN ANALYZE
SELECT h.name, COUNT(*) as completions
FROM habits h
JOIN completions c ON h.id = c.habit_id
WHERE c.completed_date >= '2025-01-01'
GROUP BY h.id;

-- Output interpretation:
-- Seq Scan = full table scan (usually bad)
-- Index Scan = using index (good)
-- Index Only Scan = index-only access (best)
-- Bitmap Index Scan = bitmap heap scan (good for many rows)
```

### Query Tips

```sql
-- BAD: SELECT *
SELECT * FROM habits;

-- GOOD: Select only needed columns
SELECT id, name, created_at FROM habits;

-- BAD: LIKE for prefix search without index
SELECT * FROM habits WHERE name LIKE '%exercise%';

-- GOOD: Prefix LIKE can use index
SELECT * FROM habits WHERE name LIKE 'exercise%';

-- BAD: Functions on indexed columns
SELECT * FROM habits WHERE lower(name) = 'exercise';

-- GOOD: Create expression index
CREATE INDEX idx_habits_lower_name ON habits(lower(name));

-- BAD: OR on different columns (hard to optimize)
SELECT * FROM habits WHERE name = 'Exercise' OR description = 'workout';

-- GOOD: Use UNION for complex OR conditions
SELECT * FROM habits WHERE name = 'Exercise'
UNION
SELECT * FROM habits WHERE description = 'workout';

-- Use CTEs for complex queries
WITH active_habits AS (
    SELECT id, name FROM habits WHERE archived_at IS NULL
)
SELECT h.name, COUNT(c.id)
FROM active_habits h
LEFT JOIN completions c ON h.id = c.habit_id
GROUP BY h.id, h.name;
```

### Efficient Date Queries

```sql
-- For DATE type
SELECT * FROM completions
WHERE completed_date >= '2025-01-01'
  AND completed_date < '2025-02-01';

-- For TIMESTAMPTZ
SELECT * FROM completions
WHERE created_at >= '2025-01-01 00:00:00+00'
  AND created_at < '2025-02-01 00:00:00+00';

-- For current month
SELECT * FROM completions
WHERE completed_date >= DATE_TRUNC('month', CURRENT_DATE)
  AND completed_date < DATE_TRUNC('month', CURRENT_DATE + INTERVAL '1 month');

-- Don't use LIKE for dates
-- BAD: WHERE completed_date LIKE '2025-01%'
```

### Table Statistics

```sql
-- Update statistics for query planner
ANALYZE habits;
ANALYZE;  -- All tables

-- Vacuum and analyze (run periodically)
VACUUM ANALYZE;
```

### Query Planning

```sql
-- Check if index is being used
EXPLAIN ANALYZE SELECT * FROM habits WHERE name = 'Exercise';

-- Force index (only when necessary)
SET enable_seqscan = OFF;  -- Use for testing only!
```

## 6. SQLAlchemy Patterns

### Engine Setup

```python
from sqlalchemy import create_engine

# Connection string format: postgresql://user:password@host:port/database
engine = create_engine(
    "postgresql://username:password@localhost:5432/habits",
    pool_size=5,           # Connection pool size
    max_overflow=10,       # Max connections beyond pool_size
    pool_pre_ping=True,    # Test connections before use
    echo=False,            # Set True for SQL logging
)
```

### Model Definition

```python
from sqlalchemy import Column, Integer, String, Text, ForeignKey, UniqueConstraint, CheckConstraint, DateTime, Boolean
from sqlalchemy.orm import relationship, declarative_base
from sqlalchemy.sql import func

Base = declarative_base()

class Habit(Base):
    __tablename__ = "habits"

    id = Column(Integer, primary_key=True)
    name = Column(String(100), nullable=False)
    description = Column(Text)
    color = Column(String(7), default="#10B981")
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    updated_at = Column(DateTime(timezone=True), onupdate=func.now())
    archived_at = Column(DateTime(timezone=True))
    is_active = Column(Boolean, default=True)

    completions = relationship(
        "Completion",
        back_populates="habit",
        cascade="all, delete-orphan",
        lazy="selectin",
    )

    __table_args__ = (
        CheckConstraint("length(name) > 0", name="name_not_empty"),
        UniqueConstraint("name", name="unique_name"),
    )


class Completion(Base):
    __tablename__ = "completions"

    id = Column(Integer, primary_key=True)
    habit_id = Column(Integer, ForeignKey("habits.id", ondelete="CASCADE"), nullable=False, index=True)
    completed_date = Column(DateTime(timezone=True), nullable=False)
    status = Column(String(10), default="completed")
    notes = Column(Text)
    created_at = Column(DateTime(timezone=True), server_default=func.now())

    habit = relationship("Habit", back_populates="completions")

    __table_args__ = (
        UniqueConstraint("habit_id", "completed_date", name="uq_habit_date"),
        CheckConstraint("status IN ('completed', 'skipped')", name="valid_status"),
    )
```

### JSON/JSONB Support

```python
from sqlalchemy import TypeDecorator, JSON

class JSONB(TypeDecorator):
    impl = JSON

    # Automatically convert Python dicts to PostgreSQL JSONB
    def process_bind_param(self, value, dialect):
        return value

    def process_result_value(self, value, dialect):
        return value

class Habit(Base):
    __tablename__ = "habits"

    id = Column(Integer, primary_key=True)
    name = Column(String(100), nullable=False)
    settings = Column(JSONB)

# Query JSONB
from sqlalchemy import cast
from sqlalchemy.dialects.postgresql import JSONB

habit = session.query(Habit).filter(
    Habit.settings['notifications'].astext.cast(Boolean) == True
).first()
```

### Relationship Loading Strategies

| Strategy | Use Case |
|----------|----------|
| `lazy="select"` | Default, N+1 if accessed |
| `lazy="joined"` | Many-to-one relationships |
| `lazy="selectin"` | One-to-many collections |
| `lazy="raise"` | Prevent accidental lazy loads |

```python
from sqlalchemy.orm import joinedload, selectinload

# Eager load in query
habits = session.query(Habit).options(selectinload(Habit.completions)).all()

# For many-to-one
completions = session.query(Completion).options(joinedload(Completion.habit)).all()
```

### Session Management

```python
from sqlalchemy.orm import sessionmaker, Session

SessionLocal = sessionmaker(bind=engine)

# Context manager pattern (recommended)
def get_habits():
    with Session(engine) as session:
        return session.query(Habit).all()

# With transaction handling
def create_habit(name: str):
    with Session(engine) as session, session.begin():
        habit = Habit(name=name)
        session.add(habit)
        return habit

# For FastAPI dependency
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

## 7. Data Integrity

### Constraints Summary

```sql
CREATE TABLE example (
    id SERIAL PRIMARY KEY,                    -- Primary key
    name TEXT NOT NULL,                        -- Required field
    email TEXT UNIQUE,                         -- No duplicates
    age INTEGER CHECK (age >= 0),              -- Value validation
    category_id INTEGER REFERENCES categories(id),  -- Foreign key
    status TEXT DEFAULT 'active'               -- Default value
);
```

### Check Constraints

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email TEXT NOT NULL,
    age INTEGER CHECK (age >= 18 AND age <= 120),
    email_verified BOOLEAN DEFAULT FALSE,
    CONSTRAINT valid_email CHECK (email ~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')
);
```

### Exclusion Constraints

```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- Prevent overlapping date ranges
CREATE TABLE reservations (
    id SERIAL PRIMARY KEY,
    room_id INT NOT NULL,
    reserved_from TIMESTAMPTZ NOT NULL,
    reserved_to TIMESTAMPTZ NOT NULL,
    EXCLUDE USING GIST (
        room_id WITH =,
        tsrange(reserved_from, reserved_to) WITH &&
    )
);
```

### Triggers

```sql
-- Function to update updated_at
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ language 'plpgsql';

-- Apply trigger to table
CREATE TRIGGER update_habits_updated_at
    BEFORE UPDATE ON habits
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();
```

### Soft Deletes

```sql
-- Instead of DELETE, set archived_at
UPDATE habits SET archived_at = NOW() WHERE id = 1;

-- Query active records
SELECT * FROM habits WHERE archived_at IS NULL;

-- Create partial index for active records
CREATE INDEX idx_active_habits ON habits(name) WHERE archived_at IS NULL;

-- Unique constraint on soft deletes (ignore archived)
CREATE UNIQUE INDEX idx_habits_name_active
ON habits(name) WHERE archived_at IS NULL;
```

## 8. Transactions

### Basic Transactions

```sql
BEGIN;
INSERT INTO habits (name) VALUES ('Exercise');
INSERT INTO completions (habit_id, completed_date) VALUES (currval('habits_id_seq'), '2025-01-15');
COMMIT;

-- On error
ROLLBACK;
```

### Savepoints

```sql
BEGIN;
INSERT INTO habits (name) VALUES ('Exercise');

SAVEPOINT sp1;
INSERT INTO completions (habit_id, completed_date) VALUES (1, '2025-01-15');
-- If this fails, we can rollback to sp1
ROLLBACK TO sp1;

-- Continue with other operations
INSERT INTO completions (habit_id, completed_date) VALUES (1, '2025-01-16');
COMMIT;
```

### Python Transactions

```python
from sqlalchemy import text

# Explicit transaction
with engine.begin() as connection:
    connection.execute(text("INSERT INTO habits (name) VALUES (:name)"), {"name": "Exercise"})
    connection.execute(text("INSERT INTO completions (habit_id, completed_date) VALUES (:id, :date)"), {"id": 1, "date": "2025-01-15"})
    # Auto-commits on success, auto-rollbacks on exception

# SQLAlchemy ORM
with Session(engine) as session, session.begin():
    habit = Habit(name="Exercise")
    session.add(habit)
    completion = Completion(habit=habit, completed_date="2025-01-15")
    session.add(completion)
    # Auto-commits/rollbacks
```

### Isolation Levels

```sql
-- Set isolation level
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;  -- Default
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

| Level | Read Uncommitted | Read Committed | Repeatable Read | Serializable |
|-------|------------------|----------------|-----------------|--------------|
| Dirty reads | Possible | Not possible | Not possible | Not possible |
| Non-repeatable reads | Possible | Possible | Not possible | Not possible |
| Phantoms | Possible | Possible | Possible | Not possible |

## 10. Performance Tuning

```ini

shared_buffers = 256MB              # 25% of RAM

# WAL settings
wal_buffers = 16MB
checkpoint_completion_target = 0.9
max_wal_size = 1GB
min_wal_size = 80MB

# Query planner
random_page_cost = 1.1              # Lower for SSD

# Connection settings
max_connections = 100
```

### VACUUM and Autovacuum

```sql
-- Manual vacuum (run during maintenance windows)
VACUUM habits;
VACUUM ANALYZE habits;  -- Update statistics too

-- Full vacuum (requires exclusive lock, use sparingly)
VACUUM FULL habits;

-- Autovacuum configuration (in postgresql.conf)
autovacuum = on
autovacuum_max_workers = 3
autovacuum_naptime = 1min
```

### Table Partitioning

```sql
-- Partition by date range
CREATE TABLE completions (
    id SERIAL,
    habit_id INT NOT NULL,
    completed_date DATE NOT NULL,
    status TEXT
) PARTITION BY RANGE (completed_date);

-- Create partitions
CREATE TABLE completions_2025_01 PARTITION OF completions
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE completions_2025_02 PARTITION OF completions
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- Benefits: Faster queries on partitions, easier maintenance
-- Can drop entire partitions instead of individual rows
```

### Materialized Views

```sql
-- Create materialized view (cached results)
CREATE MATERIALIZED VIEW habit_stats AS
SELECT
    h.id,
    h.name,
    COUNT(c.id) as total_completions,
    MAX(c.completed_date) as last_completion
FROM habits h
LEFT JOIN completions c ON h.id = c.habit_id
GROUP BY h.id, h.name;

-- Query is fast (pre-computed)
SELECT * FROM habit_stats;

-- Refresh materialized view
REFRESH MATERIALIZED VIEW habit_stats;

-- Concurrent refresh (doesn't block reads, requires UNIQUE index)
CREATE UNIQUE INDEX idx_habit_stats_id ON habit_stats(id);
REFRESH MATERIALIZED VIEW CONCURRENTLY habit_stats;
```

### Parallel Query

```sql
-- Enable parallel query
SET max_parallel_workers_per_gather = 4;

-- Check query plan with parallelism
EXPLAIN ANALYZE SELECT * FROM large_table WHERE condition = 'value';
```

## 11. Backup & Recovery

### pg_dump (Logical Backup)

```bash
# Backup entire database
pg_dump -U postgres -d habits -f habits_backup.sql

# Backup specific tables
pg_dump -U postgres -d habits -t habits -t completions -f habits_tables_backup.sql

# Custom format (compressed, allows selective restore)
pg_dump -U postgres -d habits -F c -f habits_backup.dump

# Restore from SQL file
psql -U postgres -d habits -f habits_backup.sql

# Restore from custom format
pg_restore -U postgres -d habits habits_backup.dump
```

### pg_dumpall (Cluster Backup)

```bash
# Backup all databases
pg_dumpall -U postgres -f all_databases_backup.sql

# Restore all databases
psql -U postgres -f all_databases_backup.sql
```

### Physical Backup (WAL Archiving)

```ini
# Enable WAL archiving (postgresql.conf)
wal_level = replica
archive_mode = on
archive_command = 'cp %p /path/to/archive/%f'
```

### pgBackRest (Advanced Backup)

```bash
# Full backup
pgbackrest --stanza=habits backup

# Incremental backup
pgbackrest --stanza=habits --type=incr backup

# Restore
pgbackrest --stanza=habits restore
```

### Point-in-Time Recovery

```sql
-- Create restore point
SELECT pg_create_restore_point('before_major_update');

-- Restore to specific point
recovery_target_time = '2025-01-15 14:30:00'
recovery_target_name = 'before_major_update'
```

### Backup Best Practices

- Schedule regular automated backups
- Store backups in multiple locations
- Test restore procedures regularly
- Use compression for backups
- Monitor backup success/failure
- Document recovery procedures

## 12. Anti-Patterns

### Configuration Mistakes

| Mistake | Solution |
|---------|----------|
| Default configuration | Tune based on hardware and workload |
| Insufficient shared_buffers | Set to 25% of RAM |
| High max_connections | Use connection pooling instead |
| No autovacuum | Enable and tune autovacuum |

### Schema Mistakes

| Mistake | Solution |
|---------|----------|
| Storing comma-separated lists | Use proper junction tables or arrays |
| Not indexing foreign keys | Always index FK columns |
| Using TEXT for everything | Use appropriate data types |
| Not using constraints | Enforce integrity at DB level |
| Over-normalization | Balance normalization vs performance |

### Query Mistakes

| Mistake | Solution |
|---------|----------|
| `SELECT *` | Select only needed columns |
| LIKE for date queries | Use date comparisons |
| N+1 queries | Use JOINs or eager loading |
| Not using parameterized queries | Always use placeholders |
| Not using transactions | Wrap multi-step operations |

### Python Mistakes

| Mistake | Solution |
|---------|----------|
| String formatting SQL | Use parameterized queries |
| Not closing connections | Use context managers |
| Creating engine per request | Create once, reuse |
| Ignoring connection pool leaks | Use proper pool configuration |
| Not handling exceptions | Use proper try/except |

### Common Pitfalls

```sql
-- Avoid this (schema lock!)
SELECT * FROM habits;  -- While DDL is running

-- Avoid this (unnecessary sorting)
SELECT DISTINCT name FROM habits ORDER BY name;

-- Avoid this (inefficient pagination)
SELECT * FROM large_table LIMIT 1000 OFFSET 10000;

-- Better: Use cursor-based pagination
SELECT * FROM large_table WHERE id > last_seen_id ORDER BY id LIMIT 1000;
```

### PostgreSQL Date Functions

```sql
-- Current date/time
SELECT NOW();                          -- 2025-01-15 12:30:00+00
SELECT CURRENT_DATE;                   -- 2025-01-15
SELECT CURRENT_TIMESTAMP;              -- 2025-01-15 12:30:00+00

-- Date arithmetic
SELECT NOW() + INTERVAL '1 week';      -- 1 week from now
SELECT NOW() - INTERVAL '7 days';      -- 7 days ago
SELECT DATE_TRUNC('month', NOW());     -- First of current month

-- Extract parts
SELECT EXTRACT(YEAR FROM NOW());       -- 2025
SELECT EXTRACT(MONTH FROM NOW());      -- 1
SELECT EXTRACT(DAY FROM NOW());        -- 15

-- Date formatting
SELECT TO_CHAR(NOW(), 'YYYY-MM-DD');   -- 2025-01-15
SELECT TO_CHAR(NOW(), 'Day, Month DD, YYYY');  -- Wednesday, January 15, 2025
```

### Monitoring Queries

```sql
-- Active connections

-- Table sizes

    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size

    tablename,
    indexname,

    total_time,
    mean_time
FROM pg_stat_statements
```
